# モジュール設計ガイド

このドキュメントは、`Terraform-dojo` で再利用可能なモジュールを設計・改善する際の指針をまとめたものです。  
特に、モジュール利用者が迷わないこと、破壊的変更を避けること、インターフェースが長期的に保守しやすいことを重視します。

## 1. 基本方針

- モジュールは「隠蔽」よりも「分かりやすい契約」を優先する
- 利用者が input と output だけで挙動を理解できることを目指す
- 抽象化は、重複削減よりも利用者価値が明確なときに行う
- 実装都合ではなく、利用シナリオに基づいてインターフェースを設計する

## 2. モジュールの責務

- 1 モジュール 1 責務を基本とする
- 強く結びついた複数リソースは、機能単位として 1 モジュールにまとめてよい
- AWS リソース 1 個ごとに必ずモジュール化する必要はない
- 逆に、責務が複数に分かれる場合は 1 モジュールに詰め込みすぎない

## 3. 標準ファイル構成

`modules/<module-name>/` では、基本的に以下の構成を揃えます。

- `main.tf`
- `variables.tf`
- `locals.tf`
- `outputs.tf`
- `terraform.tf`
- `README.md`

### 3.1 ファイルごとの役割

- `main.tf`: 主要なリソース定義
- `variables.tf`: 入力変数の定義
- `locals.tf`: 共通値、入力正規化、命名補助
- `outputs.tf`: 呼び出し側に返す値
- `terraform.tf`: Terraform / Provider の要件
- `README.md`: 利用例と契約の説明

### 3.2 空ファイルの扱い

- `locals.tf` は、現時点で `locals` を使っていなくても作成してよい
- `outputs.tf` も将来的に公開値を持つ見込みがあるなら先に置いてよい
- ただし、役割が曖昧な空ファイルを機械的に増やすのは避ける
- `data.tf` や機能別の分割ファイルは、必要性が出た時点で追加する

## 4. 入力設計

- すべての `variable` に `type` と `description` を定義する
- 重要な入力には `validation` を追加し、失敗を早める
- 必須 input は本当に必要なものに絞る
- 環境差分で変わらない値まで input にしない
- オブジェクト型を使う場合は、利用者が理解しやすい粒度に保つ

## 5. 出力設計

- `output` は利用者が次の処理で必要とするものだけを公開する
- 内部実装を漏らすだけの `output` は避ける
- 複数値を返す場合でも、利用者にとって意味のある単位で整理する

## 6. 互換性

- 既存の `variable` や `output` の削除・改名は破壊的変更として扱う
- デフォルト値の変更も、利用者の挙動が変わるなら破壊的変更になり得る
- 破壊的変更を行う場合は、README や変更履歴に移行手順を記載する
- 互換性を保てるなら、まず追加で表現できないかを検討する

## 7. README と使用例

- モジュールには最低限の利用例を README に記載する
- 利用例は、現実的な最小構成を優先する
- 制約事項や注意点がある場合は、利用例の近くに明記する
- 入力値が多い場合は、代表パターンを複数示してよい

## 8. 実装上の注意

- `locals` は共通化に役立つ範囲で使い、巨大な設定隠蔽には使わない
- 利用者に説明しづらい暗黙挙動は避ける
- 外部モジュールを内部で使う場合は、その採用理由と制約を明記する
- タグ、命名、ポリシーなどの共通ルールは、利用者が予測できる形に保つ

## 9. レビュー観点

モジュール変更時は、少なくとも次を確認します。

- input は過不足ないか
- output は最小限か
- 既存利用者に破壊的影響がないか
- README と実装が一致しているか
- `plan` 差分の予測性が損なわれていないか
