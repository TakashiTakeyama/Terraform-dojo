# CodePipeline（IaC）標準化の方針

Terraform で AWS CodePipeline を再利用可能にするときの**中立な原則**です。特定の組織名・プロダクト名・アカウント ID・リポジトリ名をテンプレート内に埋め込まないことを前提にします。

## 目的

- 複数プロジェクトで**同じ構造**を再利用できるようにする。
- **機密や識別子**（実在の接続 ARN、実リポジトリ、社名、環境固有の命名規則の固定値）をモジュールに閉じ込めない。
- デプロイ対象（Lambda / ECS / その他）に依存しない**骨格**と、差し替え可能な**拡張点**を分ける。

## 基本原則

### 1. 識別子はすべて入力に寄せる

モジュールのデフォルト値に次を置かない。

- 実在の AWS アカウント ID・リージョン固定の ARN
- 実在の GitHub（または他 VCS）の `owner/repo`
- 社名・サービス名を含むリソース名の固定文字列
- 実環境の CodeStar Connections / CodeConnections の ARN

代わりに **必須変数**または **呼び出し側の locals** で渡す。例示はプレースホルダのみとする（`example` / `my-app` / `000000000000` 等はドキュメント用に限定し、コードのデフォルトには使わない）。

### 2. 責務の分割

| 層 | 役割 |
|----|------|
| **パイプライン骨格** | 成果物バケット（作成または参照）、CodePipeline ロール、Source、CodeBuild ステージの連結、成果物の受け渡し |
| **ビルド・デプロイの意味** | `buildspec`（インラインまたはリポジトリ内パス）、デプロイ用の AWS API 呼び出しは **CodeBuild の IAM** で表現 |
| **承認・ゲート** | 手動承認ステージや本番限定ステージは、汎用モジュールの**オプション**または**別コンポジション**で足す |

「1 モジュールですべてのプロダクトのデプロイまで完結」より、**骨格＋呼び出し側ポリシー**の方が再利用しやすい。

### 3. IAM は最小＋合成

- **CodePipeline ロール**: S3 成果物、接続の利用、CodeBuild の起動・結果取得、（使用する場合のみ）CloudWatch Logs など、**パイプライン制御に必要な範囲**に限定する。
- **CodeBuild ロール**: S3 成果物、ログ、**そのステージが実行する API だけ**（例: Lambda の更新、ECS のサービス更新、S3 への配置）。権限の増分は **追加ポリシーを変数でマージ**する形が扱いやすい。
- 管理ポリシーを広くアタッチする設計は、**学習用スタック**と**本番再利用モジュール**を分け、後者では避けるか明示的なオプトインにする。

### 4. パイプラインの「型」を少数に揃える（A / B / C）

再利用時の迷いを減らすため、次の **3 型**を固定する。新規はまずどれに当てはまるか決め、**型が増える場合は本節と実装 README を更新**する。

| 型 | フロー（概念） | 用途の目安 | 型 A/B（アプリ系）との関係 |
|----|----------------|------------|----------------------------|
| **A** | `Source → Build`（CodeBuild 1 段） | テスト・パッケージ生成のみ。リリースは別手段 | アプリ・ライブラリの CI |
| **B** | `Source → Build → … → Build`（2 段以上） | ビルド成果物を後段で AWS に反映（Lambda / ECS 等）。段ごとに CodeBuild IAM を分けやすい | 継続的デリバリ（デプロイ段に強い権限） |
| **C** | `Source → plan（任意）→ … → apply` 等 | **インフラ定義**（Terraform 等）の plan / apply。ブランチ戦略・承認はアプリと別に設計しがち | **A/B と別系統**（同じ CodePipeline の文法を使っても、モジュール・state・`name_prefix`・リポジトリパスを混在させない） |

補足:

- **A と B の違い**は「CodeBuild ステージの本数」と「デプロイ段に渡す IAM」の有無だけ、という理解でよい。
- **C は型として独立**させる。アプリ用（A/B）の root モジュールに plan/apply 用パイプライン定義を同居させない（レビュー・権限・ライフサイクルが別物のため）。

#### このリポジトリでの実装との対応

- **型 A** … `modules/codepipeline/github-buildchain/`（`codebuild_stages` が 1 要素）。
- **型 B** … 用途ごとにラッパーを分けることを推奨: Lambda → `modules/codepipeline/pipeline-app-lambda/`、ECS → `modules/codepipeline/pipeline-app-ecs/`（いずれも内部で `github-buildchain`）。段数や役割をカスタムする場合のみ `github-buildchain` を直接呼ぶ。
- **型 C** … `modules/codepipeline/pipeline-infra-tf/`（`plan` → `apply` 固定）を **別 root・別 state・別 `name_prefix`** で呼ぶ。要件が特殊な場合は `github-buildchain` を直接呼んでもよい。

### 5. CodePipeline のバージョンと実行モード

- **V2** を前提にし、実行キュー（`QUEUED`）とソーストリガー（パスフィルタ等）を必要に応じて利用できる形にする。
- 仕様上不要なら V1 に落とさない（新規は V2 で統一）。

### 6. 成果物バケット

- **専用バケットをモジュール内で作成する**か、**既存バケット ID を必須で渡す**かを変数で明示。暗黙の命名規則に実アカウント由来の値を混ぜない。
- 暗号化キーが必要な場合は **KMS ARN / ID を変数**で渡す。

### 7. ソース接続

- GitHub を想定する場合も、接続 ARN と `FullRepositoryId`・ブランチは **すべて変数**。接続リソース自体の作成は別スタックに分けてもよい（責務の分離）。

### 8. 可観測性（任意だが推奨）

- パイプライン失敗通知は EventBridge → SNS / Chat 等を**別モジュールまたはドキュメント手順**で足せるようにし、コアモジュールを肥大化させすぎない。

## モジュール設計上のチェックリスト

実装・レビュー時に次を満たす。

- [ ] ハードコードされた ARN / アカウント / リポジトリ / 社・製品名がない
- [ ] デプロイ権限は呼び出し側から注入できる（または別ロール・別プロジェクトに分離されている）
- [ ] リソース名は `var.name_prefix` 等の**短い接頭辞**で衝突しにくくし、意味のある固定ラベルは利用者が渡す
- [ ] `README` に必須変数・最小権限の考え方・型（A/B/C）の対応表がある
- [ ] サンプルは本番と混在しない場所に分離する（本リポジトリでは `terraform/env/dev/<stack>/` 等）

## 利用側リポジトリでの置き場（推奨）

Terraform-dojo は **再利用モジュールと方針・例**の置き場であり、環境ごとの AWS リソース定義そのものをここに増やし続ける想定ではない。

**本番・検証環境でパイプラインを定義するときは、インフラ用リポジトリ側で backend（state）が分かれている Terraform root ごとに `module` ブロックを書く**のが本来の使い方である。

- **例**（チームの慣習に合わせてよい）: `terraform/env/dev/<stack>/`、`terraform/env/prod/<stack>/` … のように **環境 × スタック**単位のディレクトリを root にし、その中の `main.tf` 等から本リポジトリのモジュールを `source` で参照する。
- **入力値の置き場**は、利用側 root の **`locals.tf`（または同 root 内の `.tf`）にまとめる**運用を推奨する。`terraform.tfvars` に依存しないと、環境差分がコードレビューに載りやすい（機密のみ別経路で渡す方針と併用可）。
- **state を分ける単位**と **パイプラインの責務**を揃える。アプリ CD とインフラ CD（型 C）は **別 root・別 state**（別 `<stack>` でもよい）に置く。
- `source` は Git の Terraform-dojo を **タグやコミットでピン留め**するなど、利用側のバージョン管理ルールに従う。
- 手元検証は `terraform/env/dev/<stack>/` を参照してよいが、**本番 backend と同じ root に検証用を混在させない**。

## このリポジトリでの置き場

- **方針・レビュー基準**: `docs/guidelines/`（本書）
- **汎用コア**: `modules/codepipeline/github-buildchain/` … 任意段数の CodeBuild 連鎖
- **用途別ラッパー（推奨）**: `modules/codepipeline/pipeline-app-lambda/`、`pipeline-app-ecs/`、`pipeline-infra-tf/`（索引は `modules/codepipeline/README.md`）
- **動作確認用の例**: `terraform/env/dev/<stack>/`（`build-only` / `lambda-pipeline` / `ecs-pipeline` / `infra-pipeline`。本番は `terraform/env/prod/` に同じ構成で配置。いずれも `locals.tf` で値を定義）
